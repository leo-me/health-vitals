## 🏗️ 第二部分：技术架构 (4 分钟)

### "为什么选择这些技术栈？"

#### 后端：FastAPI

```
"我选择 FastAPI 作为后端框架，主要是：

1. 性能
   - 基于 Starlette，接近 Node.js 和 Go 的性能
   - 支持异步处理（async/await）
   - 数据密集型应用非常适合

2. 开发效率
   - 类型提示（type hints）：减少 bug
   - 自动生成 API 文档（Swagger UI）
   - 数据验证内置（Pydantic）
   - 大大减少样板代码

3. 生产就绪
   - 内置的中间件支持
   - 支持依赖注入
   - 异常处理清晰
   - 易于测试

我之前在中国的公司用过 Node.js 和 Java，
但 FastAPI 是我目前最喜欢的框架之一。
"
```

#### 数据库：PostgreSQL

```
"数据库选择 PostgreSQL，原因是：

1. 关系型数据
   - 用户、健康数据、告警规则 - 有很多关联关系
   - PostgreSQL 的 JOIN 性能非常好

2. 高级特性
   - JSONB：可以存储灵活的传感器数据
   - 全文搜索：如果需要搜索历史记录
   - 数组类型：存储多个数值的时序数据

3. 可扩展性
   - 支持分区（partitioning）
   - 支持复制和故障转移
   - 云上有托管服务（AWS RDS）

在我的架构中，数据表包括：
- users：用户信息
- health_data：时序数据（心率、体温等）
- alerts：告警规则
- alert_logs：触发的告警记录

关键是做了索引优化：
- user_id 和 timestamp 上建立复合索引
- 加快查询用户最近 7 天的数据

```

```
"前端用 React + Vite：

## 💻 第三部分：核心代码设计 (4 分钟)

### "怎么组织代码的？"

#### 分层架构

```
"我采用了标准的分层架构：

1. API 层 (routes)
   ```python
   @router.post("/health/ingest")
   def ingest_data(data: HealthDataCreate, db: Session):
       # 只负责接收请求、调用服务层、返回响应
   ```

2. 服务层 (services)
   ```python
   class HealthService:
       def process_health_data(self, raw_data):
           # 数据清洗、验证、业务逻辑
   ```

3. 数据访问层 (crud)
   ```python
   def get_health_data(db, user_id, start_date):
       # 只负责数据库查询
   ```

4. 模型层 (models)
   - ORM 模型（SQLAlchemy）
   - Pydantic Schema（数据验证）

这样做的好处：
✅ 单一职责 - 每层做自己的事
✅ 易于测试 - 可以 mock 各层
✅ 易于维护 - 改动一个层不影响其他
✅ 可扩展 - 如果要加缓存、消息队列，很容易
"
```

#### 认证和授权

```
"安全是 HealthTech 应用的关键。

实现了两个机制：

1. JWT 认证
   - 用户登录时返回 JWT token
   - 后续请求在 Authorization header 中携带
   - 服务器验证 token 的签名和过期时间

   ```python
   @app.get("/protected")
   def protected_route(current_user = Depends(get_current_user)):
       return {"user": current_user.username}
   ```

2. 权限控制 (RBAC)
   - 用户只能访问自己的数据
   - 如果要支持医生访问患者数据，需要明确授权

3. 敏感信息保护
   - 密码用 bcrypt 加密存储
   - API 不返回密码
   - 日志不记录敏感信息
"
```

#### 错误处理

```
"所有可能的错误都有处理：

1. 验证错误
   - 如果传来的 JSON 格式错误，返回 400 和清晰的错误信息

2. 业务逻辑错误
   - 用户不存在 → 404
   - 无权访问 → 403
   - 服务器错误 → 500

3. 数据库错误
   - 唯一约束违反 → 400
   - 连接失败 → 503

所有错误都有合理的 HTTP 状态码和错误信息，
方便前端处理。
"
```

这样做的好处：

✅ 可追溯 - 每个部署都有记录

## 🔄 第七部分：可扩展性讨论 (3-4 分钟)

### "如果用户量增长 10 倍，你会怎么做？"

**这是面试官经常问的一个问题。要充分展示你的系统设计思维。**

```
"很好的问题。目前系统的瓶颈是数据库。

如果用户量增长 10 倍，我会这样优化：

1️⃣ 数据库层
   - 加索引：在 user_id, timestamp 上
   - 分区：按日期分区，旧数据归档到 S3
   - 读写分离：主从复制，读从副本
   - 缓存：Redis 缓存热数据（最近 24h 的数据）

2️⃣ 应用层
   - 异步处理：数据摄入用消息队列（Celery）
   - 限流：防止恶意请求
   - 分页：大数据查询分页返回

3️⃣ 基础设施
   - 水平扩展：多个 ECS 任务实例
   - 负载均衡：AWS ELB 分散流量
   - 自动扩展：根据 CPU/内存自动增加实例

4️⃣ 监控
   - 更细粒度的监控
   - 告警规则优化
   - 性能分析和优化

具体的改进方案：

改进前：单数据库 + 单应用实例
请求流 → 应用 → 数据库
QPS 上限：~1000

改进后：缓存 + 异步 + 水平扩展
       ┌─ Redis (缓存)
请求 ──┤─ 应用 (多实例)
       └─ 数据库 (主从+分区)
QPS 上限：~10,000

这样就能轻松支持 10 倍的增长。
"
```

### "如果需要实时处理数据怎么办？"

```
"目前用的是 HTTP polling，
但如果需要真正的实时处理，我会：

1. WebSocket
   - 前端保持 WebSocket 连接
   - 服务器有新数据时立即推送
   - 更低的延迟

2. 消息队列（Kafka/RabbitMQ）
   - 从多个传感器接收数据
   - 存到消息队列
   - 多个 Worker 并行处理
   - 确保不丢数据

3. 流处理框架（Kafka Streams/Flink）
   - 实时计算窗口（1 分钟平均心率）
   - 实时告警检测
   - 聚合分析

这对于医疗应用来说很重要，
因为某些异常需要立即通知医生。
"
```

## ❓ 常见深入问题 & 回答

### Q1: "为什么不用 MongoDB？"

```
A: "MongoDB 是 NoSQL，适合无结构或半结构的数据。
   但我们的数据很有结构：
   - 用户信息：用户名、邮箱、密码（结构化）
   - 健康数据：虽然可能字段很多，但都是已知的传感器类型
   - 告警规则：有明确的约束（>100, <50 etc）
   
   用 PostgreSQL 的 JSONB 也能处理灵活字段，
   同时还能用 SQL 的强大功能。
   
   而且，在 HealthTech，关系数据库是行业标准，
   合规性也更好。"
```

### Q2: "怎么保证数据隐私？"

```
A: "这是 HealthTech 最关键的问题。我做了：

1. 端对端加密（理想情况）
   - 但实现复杂，我的 MVP 没做

2. 传输加密
   - 所有通信都用 HTTPS
   - TLS 1.3

3. 存储加密
   - PostgreSQL：pgcrypto 加密敏感字段
   - S3：启用服务端加密

4. 访问控制
   - 用户只能看自己的数据
   - 医生需要明确授权才能看患者数据
   - 所有访问都有日志

5. GDPR 合规
   - 支持数据导出
   - 支持数据删除
   - 隐私政策清晰

在生产环境中，还需要：
- 定期安全审计
- 渗透测试
- 合规性认证（HIPAA for US, GDPR for EU）
"
```

### Q3: "用什么工具监控这个系统？"

```
A: "用 AWS CloudWatch：

1. 日志
   - 应用日志：错误、警告、信息
   - 访问日志：谁访问了什么
   - 自动转发到 CloudWatch

2. 指标
   - 响应时间（P50, P95, P99）
   - 错误率
   - 数据库查询时间
   - CPU、内存使用率

3. 告警
   - 错误率 > 5% → 发告警
   - 响应时间 > 1s → 发告警
   - 磁盘空间 < 10% → 发告警

4. 仪表板
   - 自定义仪表板展示关键指标
   - 可以看到系统的实时状态

对于 HealthTech 应用，监控特别重要，
因为系统宕机可能影响患者安全。"
```

### Q4: "如果一个请求特别慢，怎么排查？"

```
A: "我会用这个流程：

1. 看 CloudWatch 日志
   - 是前端慢、网络慢、还是后端慢？

2. 如果是后端
   - 看响应时间分布
   - 找出最慢的 API

3. 性能分析
   - 是数据库查询慢？用 EXPLAIN ANALYZE
   - 是应用逻辑慢？用 Python profiler
   - 是网络慢？看延迟

4. 优化方案
   - 慢查询：加索引、重写 SQL、分页
   - 慢逻辑：优化算法、加缓存
   - 网络：CDN、压缩

5. 验证
   - 部署优化后，对比性能改进

这种工程思维是我在大厂学到的。
数据驱动，而不是猜测。"
```

## 💡 面试小技巧

### ✅ 要做的

1. **讲故事，不要背稿**
   - 自然流畅的对话比完美背诵更好

2. **用数据说话**
   - "性能提升 50%"比"很快"好
   - "处理 1000 QPS"比"能处理很多请求"好

3. **展示思考过程**
   - "我为什么选择 A 而不是 B"
   - 面试官想看你怎么做决策

4. **诚实承认不足**
   - "这部分我没优化，因为..."
   - 比装懂更好

5. **反问面试官**
   - "这家公司面临什么技术挑战？"
   - 显示你的好奇心

### ❌ 不要做的

1. ❌ 过度技术细节
   - 面试官不需要听 SQL 执行计划
   - 聚焦于"为什么"而不是"怎么做"

2. ❌ 贬低其他技术
   - "Node.js 很垃圾"
   - 展示尊重，不同技术适合不同场景

3. ❌ 夸大其词
   - 说谎会被识别出来
   - "我能处理百万级 QPS"而实际没测过

4. ❌ 沉默
   - 面试是对话，不是演讲
   - 主动解释，邀请问题
